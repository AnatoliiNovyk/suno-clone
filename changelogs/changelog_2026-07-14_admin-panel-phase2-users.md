# changelog_2026-07-14_admin-panel-phase2-users

## Зміна
Фаза 2 адмін-панелі — розділ «Користувачі» з повноцінним CRM-інтерфейсом:

- **`/admin/users`** (`AdminUsersPage.tsx`) — список профілів із серверним пошуком (email/`display_name`, debounce 300 мс), фільтрами-чіпами за планом і роллю, пагінацією по 25 записів (`range()` + `count: 'exact'`), сортуванням за `created_at` DESC. Клік рядка → картка користувача.
- **`/admin/users/:id`** (`AdminUserDetailPage.tsx`) — паралельне завантаження профілю, 30 останніх операцій із `credit_transactions` і 20 останніх треків. Секції: профіль (email, імʼя, дата, поточні баланс/тариф/роль), таблиця операцій (кольори `success`/`error` за знаком суми), компактний список треків.
- **Три модалки в `src/components/admin/`** — `AdjustCreditsDialog` (числова зміна балансу + обовʼязкова причина), `SetPlanDialog` (селект активних планів), `SetRoleDialog` (`user`/`admin`; UI-блокування самодепромоції) — усі викликають наявні SECURITY DEFINER RPC (`admin_adjust_credits`/`admin_set_plan`/`admin_set_role`) через `supabase.rpc(...)`. Спільний примітив `AdminActionDialog` (без нових залежностей) + мапер `rpcErrorMessage()` для повідомлень про `forbidden` / `insufficient_credits_or_missing_user` / `cannot_demote_self` / `unknown_plan` / `invalid_role` / `reason_required`.
- Каркас: `AdminLayout.tsx` активує пункт «Користувачі» (`ready: true`); `App.tsx` додає два вкладені маршрути `users` і `users/:id`.
- Оновлено секцію Admin panel у `CLAUDE.md`.

## Як було до зміни
Фаза 1 (PR #13) заклала базу безпеки — RLS + RPC, — але єдиний робочий розділ адмінки був дашборд. Пункт «Користувачі» в сайдбарі був позначений «скоро»; будь-яке керування балансом, тарифом чи роллю робилось вручну через SQL Editor, без пошуку/фільтрів/пагінації і без інтерактивного контексту (історія операцій, треки користувача).

## Що покращує зміна
- Адмін працює з користувачами через звичний CRM-екран, не пише SQL для рутинних дій.
- Усі привілейовані дії йдуть через ті самі RPC, що й раніше — з перевіркою `is_admin()` і записом у `admin_actions`. Фронтенд лише зручна оболонка: підроблений клієнт без ролі отримає `forbidden` від БД.
- UI-подвоєння серверної заборони самодепромоції (`cannot_demote_self`) робить неможливий сценарій ще й неможливим на кнопці — менше плутанини, коротший шлях до правильної поведінки.
- Основа готова до фаз 3–5 (треки/тарифи/підписки/аудит) — спільний `AdminActionDialog` і мапер помилок будуть перевикористані.
