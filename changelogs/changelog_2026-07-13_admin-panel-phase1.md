# changelog_2026-07-13_admin-panel-phase1

## Зміна
Фаза 1 адмін-панелі (CMS): фундамент безпеки в БД + каркас UI з дашбордом.

- **Міграція `supabase/migrations/1784150400_admin_panel_foundation.sql`** (+ ті самі об'єкти в `bootstrap.sql`): таблиця аудиту `admin_actions` (RLS: читають лише адміни, пишуть лише SECURITY DEFINER функції); admin-RLS політики (SELECT усіх `profiles`/`tracks`/`subscriptions`/`credit_transactions`, UPDATE/DELETE будь-якого треку, повний CRUD `plans`/`plan_prices`); RPC `admin_adjust_credits(user, delta, reason)` (обов'язкова причина, запис у `credit_transactions` + аудит), `admin_set_plan`, `admin_set_role` (заборона знімати роль із себе).
- **Фронтенд**: `AdminRoute` (guard за `user.role`), `AdminLayout` (сайдбар: Дашборд активний, Користувачі/Треки/Тарифи/Підписки/Аудит — позначені «скоро» до фаз 2–5), сторінка `AdminDashboardPage` (метрики: користувачі, треки за статусами, витрачені кредити за 7/30 днів, активні підписки; CSS-бар-чарт генерацій за 14 днів без нових залежностей). Маршрут `/admin` у `App.tsx`, пункт «Admin» у хедері для адмінів.
- `CLAUDE.md`: новий розділ «Admin panel».

## Як було до зміни
Після видалення мерчантів у проєкті не існувало жодного адмін-інтерфейсу: `profiles.role` та `is_admin()` були «сплячою» інфраструктурою. Будь-яка адміністративна дія (подивитись користувачів, нарахувати кредити, змінити план) вимагала ручного SQL у Dashboard, без сліду в аудиті.

## Що покращує зміна
- Адміністратор бачить стан платформи на дашборді `/admin` і має захищені інструменти для дій: усі привілейовані операції проходять через SECURITY DEFINER RPC із перевіркою `is_admin()` всередині БД — навіть підроблений клієнт без ролі отримає `forbidden`.
- Кожна адмін-дія залишає слід у `admin_actions` (хто, що, над ким, деталі) — основа для журналу аудиту фази 5.
- Каркас (guard + лейаут + маршрути) готовий до нарощування фаз 2–5 без переробок.
